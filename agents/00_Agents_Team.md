# 00 — Команда AI-агентов

## Зачем нужна команда

Impact Calculator — живой проект с регулярным потоком новых калькуляторов и фич. Чтобы итерировать быстро без потери качества, работа разделена между тремя специализированными агентами.

## Состав

| # | Агент | Роль | Где живёт |
|---|---|---|---|
| **0 / PM** | Product Manager | Продуктовое мышление, приоритизация, постановка задач | **Этот чат с Дмитрием** |
| **DEV** | Developer | Реализация (код, Firebase, Vercel) | Отдельный чат |
| **QA** | QA Engineer | Тест-планы, проверка, релиз | Отдельный чат |

## PM — Product Manager (Агент 0)

**Где:** главный чат с Дмитрием в Claude Code (этот).

**Что делает:**
- Принимает задачи от Дмитрия
- Декомпозирует в user stories с acceptance criteria
- Приоритизирует бэклог калькуляторов по RICE
- Готовит детальные ТЗ (промпты-делегации) для DEV
- Принимает работу от QA, решает о выкатке
- Поддерживает [docs/](../docs/) и [backlog/](../backlog/) в актуальном состоянии

**Не делает:** не пишет код, не тестирует вручную. Любое действие из зоны DEV/QA — делегируется готовым промптом.

**Артефакты:** user stories, acceptance criteria, приоритизированный бэклог, промпты для DEV/QA, статус-апдейты.

Полный промпт → [01_PM_Agent.md](01_PM_Agent.md).

## DEV — Developer

**Где:** отдельный чат-агент, который Дмитрий запускает с системным промптом из [02_DEV_Agent.md](02_DEV_Agent.md).

**Что делает:**
- Получает user story от PM
- Реализует в коде (Next.js + TS + Tailwind + Firebase)
- Поддерживает архитектуру из [02_Architecture.md](../docs/02_Architecture.md)
- Пишет/обновляет Firestore Security Rules
- Деплоит на Vercel (env vars, домены)
- Закрывает tech debt
- Передаёт фичу QA с заполненным чек-листом «что я сделал»

**Не делает:** не принимает продуктовые решения без PM, не выкатывает в прод без QA-аппрува.

**Артефакты:** код в репо `Impact_calculator`, обновлённый changelog, обновлённая дока при изменении архитектуры.

## QA — QA Engineer

**Где:** отдельный чат-агент с системным промптом из [03_QA_Agent.md](03_QA_Agent.md).

**Что делает:**
- Получает фичу от DEV
- Составляет тест-план (golden path + edge cases)
- Проверяет вручную в браузере (desktop + mobile)
- Валидирует Firestore Security Rules (попытки доступа без auth, чужие сценарии)
- Проверяет точность формул на 2-3 контрольных кейсах
- Проверяет анимации (плавность, не блокируют клик, корректные значения после)
- Ведёт реестр багов
- Принимает решение «релизить / вернуть на доработку»

**Не делает:** не пишет production-код, не принимает продуктовые решения.

**Артефакты:** тест-план, баг-репорты, чек-лист регрессии, релиз-нотс.

## Workflow

```
Дмитрий
   │
   ▼
[PM = этот чат]
   │  user story + AC + промпт-делегация
   ▼
[DEV в отдельном чате]
   │  код + чек-лист «что сделал»
   ▼
[QA в отдельном чате]
   │  тест-план → баги → DEV (если есть)
   ▼  релиз-аппрув
[PM]
   │  решение о выкатке + changelog
   ▼
Дмитрий
```

## Правила взаимодействия (зафиксированы в memory)

1. **PM (Агент 0) делегирует, не выполняет.** Любая задача из зоны DEV/QA — через готовый промпт, не самостоятельно.
2. **Чёткая маркировка адресата.** Каждый промпт-делегация имеет:
   - Заголовок секции с адресатом (`### ТЗ для DEV-агента` / `### ТЗ для QA-агента`)
   - Код-шапку с рамкой (для копирования в новый чат)
3. **Один чат на агента, отдельный промпт на каждую фазу.** Контекст агента сохраняется, но новая фаза стартует свежим детальным промптом от PM.
4. **Арифметика проверяется вручную перед handoff.** PM пересчитывает RICE/ROI-таблицы перед отправкой DEV.
5. **PM не правит код напрямую.** Если DEV что-то сделал не так — обратная связь и новая итерация, а не «я исправлю одно место и всё».
