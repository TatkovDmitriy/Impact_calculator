## Операционные ритуалы PM

### Старт нового калькулятора

1. Получить запрос от Дмитрия
2. Уточнить: какой бизнес-вопрос решает? кто смотрит результат (CPO / сам Дмитрий)?
3. Найти источник данных в Google Sheets → зафиксировать в [docs/05_Metrics_Source.md](../docs/05_Metrics_Source.md)
4. Записать спеку в [docs/06_Calculators_Catalog.md](../docs/06_Calculators_Catalog.md)
5. Добавить в [backlog/00_Backlog.md](../backlog/00_Backlog.md) с RICE, проверить арифметику
6. Если P0/P1 — сформировать ТЗ для DEV включая требования к аналитике

### Дизайн-ревью (после каждого DEV-деплоя на Preview)

Чеклист:
- [ ] Фон navbar = #2F3738, акцент = #FDC300, нет зелёного
- [ ] Шрифт = Manrope (или LM Main fallback)
- [ ] Логотип присутствует в navbar
- [ ] Числа в калькуляторе читаемы без скролла (fold-check)
- [ ] Кнопка "Сохранить сценарий" видна до прокрутки
- [ ] Графики корректно отражают расчёт из формул
- [ ] На мобиле (375px) ничего не ломается горизонтально

### Онбординг агента (DEV или QA)

1. Отправить онбординг-промпт из [99_Onboarding_Common.md](99_Onboarding_Common.md)
2. Дождаться ответа агента с 3 вопросами (признак что прочитал)
3. Ответить на вопросы — дать зелёный свет
4. Только после этого отправить Phase 1 промпт

### Приём работы от QA

1. Прочитать релиз-аппрув
2. Если открытые баги — оценить критичность: вернуть DEV (критический) или принять known-issue (косметический)
3. Если ок — обновить changelog и статус в [docs/01_Overview.md](../docs/01_Overview.md)
4. Дмитрий принимает финальное решение о merge в main

### Раз в спринт (~2 недели)

- Пересмотреть RICE в бэклоге, уточнить приоритеты
- Сверить данные Sheets — обновились ли метрики
- Проверить что трекинг событий работает (через Firebase Analytics dashboard)
- Сверить с роадмапом стратегии ЛП в Obsidian
